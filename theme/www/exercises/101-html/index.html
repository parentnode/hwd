Exercise 101 - HTML text

This exercise is dealing with very basic semantic HTML content formatting, using only heading (<h1>, <h2> and <h3>), paragraph (<p>), a (<a>), strong (<strong>) and emphasize (<em>) tags.

There are two purposes of this exercise: To write HTML and to practice deducting the semantic structure of an unformatted text.

Tasks

  1. Get the materials ready

    The materials for this exercise if found in the folder exercises/101-html/101. Copy the 101 folder to your personal homework folder in homework. You created your personal homework folder before you started the exercises.

  2. Add HTML to the text in 101/index.html

    The structure and hierarchy of the text needs to be reflected by html. You will need to read the text to identify the hierarchy of the text.

  3. Add a tags for the links

    Links in the text are marked in [xxx]. The links should be applied to the paragraph just before the brackets, using the value inside the brackets as the href property.

  4. Add strong and emphasis tags

    Certain words in the sentences could use some additional markup because the are particularly important or needs emphasis.

  5. Check your HTML formatting

    Make sure you opened and closed all tags appropriately.

  6. Validate your HTML on w3c

    Use https://validator.w3.org/#validate_by_input to validate your HTML.

  7. Check the document outline

    Use https://gsnedders.html5.org/outliner/ to a get a structural representation of your HTML.

Do's and don'ts

Do's

Organize your our html, for you, people and the search engines

It is important that your html is organized in such a way, that it improves your search rank on google. The "higher up" your page gets on google the better is the chance that people find it. At the same time its important for people to get an overview when they look at your page, and last but no least you also need to keep an overview of your articles headings paragraphs and subheadings. Read more about html structure why its important and how to do it here on structure.

Example good structure and good indentation:

<div class="article animals"> <h1>Animals living on earth</h1> <p>intro text</p> <h2>Habitats</h2> <p>intro text block</p> <h3>Cold climate</h3> <p>info text block</p> <h3>Warm climate</h3> <p>intro text</p> <h4>tropical</h4> <p>info text block</p> <h4>subtropical</h4> <p>info text block</p> <h2>Types of animals</h2> </div>

Note that 1 indentation is normally the size of 4 "spaces" or 2 "spaces", but we use the size of 4 "spaces", if you press the tab button (the one next to the letter Q) you can make an indentation.

Note also that there is only one heading h1 in the example above. That is a general rule, you only have one h1 per article. So if you for example have 5 pages and you have one article on each page, you should have 5 h1's in total on the entire website.

Note that the div start and end tag are above each other. This is a way to show us, the coder, that the all the "stuff" in between is in fact "inside of it". When an element has an inner element "inside of it the "inner element" is the child and the "outer" is a parent to that child. So our div here has a lot of children or childnodes as they are also called. So remember indenting elements doesn't make them children of things. it's when the tags are inside each other they become children. Just like the a, or em tag is often a child of a p tag. So all all the tags inside of our div example are actually siblings to each other, since non of them have any children in this example. They all have the same parent div, and the div have an other parent we can't see here in our example. At some point there is a parent called a body and later a final parent i the html tag, that wraps around everything in out html document, as you see in the exercise.

Tips for good headings

Headings are a very important for search engines, and people, especially h1. Make sure the headings reveal as much information as possible, the headings are not a teaser, confusing or abstract. The headings should be precise and descriptive and not to short. the worst kind of headings are misleading, and not describing at all what is really going on in the article. "Click bait". or bad news papers headings are a good example of bad headings. Good headings are so good that you get a feeling you already know exactly what the rest of the article or sub article is about. The search algorithm will most likely reward you for this kind of honest behavior. So think twice about what you write in a heading.

Pay attention to "white space"

White space is newline's between paragraph or headings ect. it can actually even be the Make sure that the "white space" makes sense and there is a consistent system behind it. Often It can make sense to leave on line of white space above a heading to make it more readable. Remember to delete all the white space that dosen't have a purpose.

Starting to think about arranging html elements for styling

This is a more complex task that we will get more into in later exercises. For now you need to know that all elements (divs, headings, paragraph ect) can be moved around pretty much as you please later with css. Every start and end tag and also image tags makes up what is called an element. These are all blocks that takes up space in a form of a square on the screen. Some elements like divs are called containers and are used for keeping things inside of. These are very handy for styling when we get to css. You don't need to use them now you just need to know they exists, read more about div's You can see where all the boxes are by using the inspector, see short video from Kahn Academy. The inspector will be your workhorse if you are gonna make any website. So you might as well get to meet it. But just keep in mind that the html has very little to do with the actual design. In other words:

The html is not the design!

Other good ideas

Use the source code of this page as inspiration for how to write html (right click -> view page source) you might even find mistakes.

Make sure you can understand the code if you come back to it in a year. Comments, readme or personal notes can be good for this.

If you do write comments, write it above the code you are talking about not next to or below it.

Try to write as few and short comments as possible if you really need them at all. When you change the code you have to change the comments as well, otherwise you will get lost. So fewer comments means less maintenance. But again if nobody else gets it, it need comments. Keep the bus-factor low. always ask yourself: If I was hit by a bus tomorrow would anyone be able to read my code and maintain it?

If things are related you can "glue" them together by not making white space

Don'ts

  • Don't replace a heading with a strong tag
  • Don't organized your html the way you want it to look on the page, wait untill you learn css
  • Don't make unnecessary white space
  • Don't make bad headings, they needs to give a clear overview, for you and the internet

  • Don't just use a tag because you feel like using it, it need's to make sense.

  • don't cheat, you know what that means, copy paste ect

Make sure you don't leave space between the tag and the words like so:

<p> Don't leave space before/after a p-tag like here in this paragraph. </p> <p>This is the way to do it.</p> This also count for h1,h2, em, spans, a, anywhere where there is text.

Don't use strong instead of a h2 or h3, a lot of websites do this, but that dosent make it right. Remember we use h1,h2,3 for search optimization, and structure.

One could argue that arranging the example with the div in such a a way the each subheading is indented reletively to the one before like this: Dont do this: <h2>heading</h2> <h3>sub heading</h3> But dont do it. It will get confusing when you have a lot of html.
